# Инструкция пользователя SaleMaker

## Назначение
**SaleMaker** — приложение для заполнения реестра в Excel по данным из справочников, заполнения чек‑листа и привязки фотографий к записям.  
Результат работы — **файл реестра Excel** и при необходимости **ZIP‑пакет** для отправки (реестр + фотографии).

Интерфейс построен на **PySide6 (Qt)** — для работы требуется установка зависимостей из `requirements.txt` (включая `PySide6`).

---

## Важные принципы (как устроены данные)
- **Все поля трактуются как текст**: при чтении Excel приложение старается читать данные как строки, чтобы не появлялись значения вида `28.0`.
- **Позиционная запись в Excel предпочтительнее**: для большинства полей можно указать `excel_column` (буква/номер столбца). Это надёжнее, чем привязка к названию заголовка.
- **`start_row` влияет на навигацию**: индекс «Запись № N» считается от первой строки данных `template.start_row`.

## Интерфейс: что где находится

### Верхняя шапка
- Название **SaleMaker**
- Дополнительное ключевое поле (настраивается в `ui.header_field`)

### Панель «Серийный номер и поиск в справочнике» (видна на любой вкладке)
- **Серийный номер \*** и кнопка **«Найти в справочнике»**
- **Номер по п/п на устройстве** и кнопка **«Найти по п/п»**
- Справа: **индикатор текущей записи/счётчик** (например, «Запись № 3 из 50»)

### Навигация (видна на любой вкладке)
- **← Предыдущая запись / Следующая запись →**
- **Новая запись** (работает как «Записать в реестр»: выполняет сохранение и переводит в новую запись)
- **Сохранить изменения** (обновляет текущую строку реестра)
- **Удалить текущую запись**
- Справа:
  - **Записать в реестр**
  - **Сформировать пакет на отправку** (самая правая)

### Вкладки
- **Общие настройки и справочник**
- **Единица оборудования**
- **Чек‑лист**
- **Фотографирование**
- **Редактор конфигурации** — правка блоков `config.yaml` (чек‑листы, `default`, semantic и др.; не файл шаблонов по модели)

## Быстрый старт (типовой сценарий)
1. Откройте вкладку **«Общие настройки и справочник»**.
2. Укажите пути:
   - `config.yaml`
   - **Справочник Excel** → нажмите **Открыть**
   - (рекомендуется) **Справочник цен** → **Открыть** — списки производителя/модели и автоподстановка типа оборудования
   - **Шаблон Excel** (шаблон реестра)
   - **Файл реестра** (куда писать строки)
   - **Папка фото**
3. В верхней панели введите **Серийный номер** (обязательно).
4. Нажмите **«Найти в справочнике»** (или поиск по п/п).
5. Проверьте/дозаполните поля на вкладке **«Единица оборудования»**.
6. Заполните вкладку **«Чек‑лист»** (при выборе типа оборудования подставятся значения `default` из конфигурации; при выборе модели — шаблон из `checklist_templates.yaml`, если он сохранён).
7. На вкладке **«Фотографирование»** включите камеру, при необходимости включите **«Фото с превью»** и сделайте фото по слотам.
8. Нажмите **«Записать в реестр»** (справа в навигации) — данные запишутся и приложение перейдёт к новой записи.

## Запись в реестр: как это работает

### «Записать в реестр»
- Если выбрана **существующая запись реестра** — выполняется **обновление текущей строки**, затем переход в режим **новой записи**.
- Если вы уже в режиме **новой записи** — выполняется **добавление новой строки**, затем переход в режим **новой записи**.

### «Новая запись»
Работает **аналогично «Записать в реестр»**.

### «Сохранить изменения»
Записывает значения полей **в текущую строку** реестра (обновление).

### Проверка уникальности
- Уникальность серийного номера и номера п/п проверяется **относительно текущей позиции**:
  - при обновлении текущей строки приложение игнорирует текущую строку (если запись одна — обновление разрешено)
  - при добавлении новой записи уникальность проверяется по всему реестру

### Защита от потери изменений при навигации
Если вы изменили поля и нажали навигацию, приложение предложит:
- **Да** — сохранить изменения
- **Нет** — не сохранять (значения будут восстановлены из реестра)
- **Отмена** — остаться на текущей записи

## Справочники

### Справочник Excel (основной)
Используется для подстановки данных по серийному номеру или по номеру п/п.

Индикатор состояния:
- поле пути **серое**, пока справочник не открыт
- после успешного **Открыть** — обычный цвет

### Справочник цен (Excel)
Используется для полей **Производитель** и **Модель** в чек‑листе (если в конфигурации заданы поля с `id: manufacturer` и `id: model`).

| Поле | Источник значений |
|------|-------------------|
| **Производитель** (`manufacturer`) | Список уникальных **Вендор** из справочника цен |
| **Модель** (`model`), производитель **не выбран** | Модели из файла **`checklist_templates.yaml`** для текущего **типа оборудования** (список обновляется при открытии выпадающего списка) |
| **Модель** (`model`), производитель **выбран** | Модели выбранного вендора из **справочника цен** |

Дополнительно:
- при заполненных **производителе** и **модели** автоматически подставляется **Тип оборудования** по колонке **Наименование** справочника цен;
- если тип оборудования изменён автоматически — чек‑лист **перестраивается** под новый тип;
- ручная смена типа оборудования не сбрасывается при последующих действиях с производителем/моделью (кроме явной автоподстановки по паре вендор+модель из справочника).

Настройки в `config.yaml`:

```yaml
price_reference:
  path: "C:/path/to/prices.xlsx"
  sheet: "Справочник"
  header_row: 3   # строка заголовков (1-based)
```

## Чек‑лист

### Два уровня подстановки значений

| Уровень | Где хранится | Когда применяется | Как править |
|---------|--------------|-------------------|-------------|
| **Значения по умолчанию для типа** | `config.yaml` → `checklists.<тип>.fields[].default` | При выборе **типа оборудования** (форма пересобирается) | Вкладка **«Редактор конфигурации»** или внешний редактор YAML |
| **Шаблон для модели** | `checklist_templates.yaml` (отдельный файл) | При выборе **модели** (поиск по паре «тип + модель») | Кнопки **«Использовать как шаблон»** / **«Очистить шаблон»** на вкладке «Чек‑лист» |

Шаблоны **не записываются** в `config.yaml`. Значения `default` в конфигурации **не изменяются** кнопками шаблона.

### Выпадающие списки + ввод своего значения
Для текстовых/числовых полей чек‑листа можно задать список `values`. Тогда в UI будет список, но можно вводить своё значение.

Пример:

```yaml
fields:
  - id: manufacturer
    label: "Производитель"
    type: text
    values: ["HP", "Dell", "Lenovo"]
    default: "HP"
    required: true
```

### Обязательные поля
Если у поля стоит `required: true`, в UI рядом с подписью будет `*`.  
Перед записью в реестр приложение проверит, что значение не пустое (после `strip()` длина ≥ 1).

Где поддерживается `required`:
- `semantic[*].required`
- `checklists.<тип>.fields[*].required`
- `column_mapping.<поле>.required` (расширенный формат)

### Один чек‑лист для нескольких типов оборудования (aliases)
Можно задать, что один и тот же чек‑лист применяется для нескольких значений «Тип оборудования»:

```yaml
checklists:
  "ПК":
    aliases: ["ПК", "Ноутбук", "Моноблок", "Неттоп"]
    checklist_start_column: 57
    fields: ...
    photo_slots: ...
```

### Динамические значения в `values` (подстановка из текущих полей)
В `fields[].values` можно использовать шаблоны вида `{...}`, которые будут подставляться из текущих значений полей.

Пример: подставить найденный/вычисленный **Тип оборудования** в список значений:

```yaml
- id: some_field
  label: "Поле"
  type: text
  values:
    - "{equipment_type}"
    - "Другое значение"
```

Доступные переменные:
- `{equipment_type}` — текущее значение поля **Тип оборудования**
- `{serial}` — серийный номер
- `{device_pp}` — номер по п/п
- также доступны:
  - значения `semantic` по `key`
  - значения «данных из справочника» по названию поля (ключ в `column_mapping`)

### Значения по умолчанию (`default`) и пустые поля
- Если у поля задано непустое `default`, при построении чек‑листа оно **подставляется автоматически**.
- Если указано `default: null` или значение пустое — поле остаётся **пустым** (выпадающий список без выбранного пункта).
- В `default` допустимы шаблоны `{equipment_type}`, `{serial}` и др. (см. ниже).

Пример «без значения по умолчанию»:

```yaml
- id: ram
  label: "ОЗУ"
  type: text
  default: null
```

### Динамические значения в `default` (подстановка при построении формы)
В `fields[].default` можно использовать шаблоны `{...}`.  
Если у поля есть явное непустое `default`, приложение подставит вычисленное значение при построении/обновлении чек‑листа.

Пример:

```yaml
- id: comment
  label: "Комментарий"
  type: text
  default: "{equipment_type} / SN: {serial}"
```

Доступные переменные — те же, что для `values` (см. раздел **«Динамические значения в `values`»** выше).

### Шаблоны чек‑листа по модели (файл `checklist_templates.yaml`)

**Шаблоны по модели** хранятся в **отдельном файле** (по умолчанию `checklist_templates.yaml` в той же папке, что и `config.yaml`). Путь задаётся в конфигурации:

```yaml
checklist_templates:
  path: ''   # пусто — checklist_templates.yaml рядом с config.yaml
```

Файл шаблонов **не редактируется** через вкладку «Редактор конфигурации» — только кнопками на вкладке «Чек‑лист» или вручную в текстовом редакторе.

#### Формат файла

```yaml
templates:
  - equipment_type: ПК
    model: "Latitude 5520"
    fields:
      manufacturer: "Dell"
      cpu: "Intel Core i5"
      ram: "16 ГБ"
```

- **`equipment_type`** — тип оборудования (как в поле «Тип оборудования»).
- **`model`** — ключ поиска; в `fields` **не дублируется**.
- **`fields`** — значения полей чек‑листа по `id` (включая **производителя**; модель задаётся только ключом записи).

#### Поведение в интерфейсе

1. **Выбор типа оборудования** — поля заполняются только из `default` в `config.yaml`.
2. **Выбор модели**:
   - если загружен справочник цен и указаны производитель и модель — может автоматически определиться тип по колонке **Наименование**, затем поиск шаблона;
   - если производитель пуст — модель можно выбрать из списка шаблонов для текущего типа (список обновляется при открытии выпадающего списка);
   - при совпадении «тип + модель» значения из `fields` подставляются в форму (модель не перезаписывается);
   - в строке состояния: **«Найден шаблон для модели …»** или сообщение, что шаблон не найден.
3. **Загрузка записи из реестра** — значения из Excel **не перезаписываются** автоподстановкой шаблона.
4. **Использовать как шаблон** — сохраняет текущие заполненные поля (кроме модели) в файл шаблонов для текущих типа оборудования и модели. Модель должна быть указана.
5. **Очистить шаблон** — удаляет запись «тип + модель» из файла шаблонов и пересобирает чек‑лист по `default` из `config.yaml`.

> Чтобы модель появилась в выпадающем списке **без выбранного производителя**, её нужно хотя бы раз сохранить кнопкой **«Использовать как шаблон»** для данного типа оборудования.

#### Кнопки внизу вкладки «Чек‑лист»
- **Использовать как шаблон**
- **Очистить шаблон**

## Фотографирование

### Камера
- Можно выбрать камеру из списка (индексы устройств).
- Есть кнопка **«Обновить»** список камер.
- На Windows при проблемах с камерой приложение перебирает несколько backend'ов (DirectShow, Media Foundation).

### Съёмка с превью на весь экран
На вкладке **«Общие настройки и справочник»** (и в `config.yaml`) доступна опция:

- **«Фото с превью (полный экран перед съёмкой…)»** — перед сохранением кадра открывается полноэкранное окно с живым изображением камеры.
- **Пробел** или **щелчок** — сделать снимок; **Esc** — отмена без сохранения.

```yaml
camera:
  max_width: 0
  max_height: 0
  scan_max_index: 5
  capture_with_preview: true
```

Если `capture_with_preview: false`, снимок сохраняется сразу из превью на вкладке «Фотографирование».

### Авто-кадрирование центрального объекта
При включённой опции после съёмки кадр **обрезается** по контуру основного объекта в центральной зоне; в файл сохраняется только выделенная область (с небольшими полями по краям).

- На вкладке **«Общие настройки и справочник»**: **«Авто-кадрирование: сохранять только центральный объект…»**
- В `config.yaml`:

```yaml
camera:
  crop_central_subject: false
  crop_padding_ratio: 0.05      # отступ вокруг объекта (доля ширины/высоты bbox)
  crop_center_region: 0.75      # зона поиска объекта (доля кадра по центру)
  crop_min_area_ratio: 0.01     # мин. площадь объекта (доля кадра)
  crop_max_area_ratio: 0.95     # макс. площадь объекта (доля кадра)
```

Рекомендации: объект по центру кадра, **контрастный однотонный фон** (фон должен быть виден по краям кадра), достаточное освещение. Если объект не найден, сохраняется **полный кадр** без обрезки.

**Индикация в интерфейсе:** при включённом авто-кадрировании на превью камеры и в полноэкранной съёмке рисуется **зелёная рамка** — это область, которая будет сохранена в файл. Под превью камеры или внизу окна съёмки выводится текст: объект найден / не найден. После сохранения — сообщение в строке состояния.

### Максимальное разрешение съёмки
Настраивается в **«Общие настройки и справочник»** и сохраняется в `config.yaml`:

```yaml
camera:
  max_width: 0
  max_height: 0
  scan_max_index: 5
```

Если `max_width/max_height = 0`, разрешение не задаётся (берётся стандартное от камеры).

> Пути к файлам фото с кириллицей поддерживаются (сохранение через кодирование в память, без ограничений `imwrite` OpenCV).

### Фото‑слоты
Для каждого слота:
- кнопка **«Снять»**
- поле **«Имя файла»** (это попадёт в Excel)
- подсказка `hint` (если задана в конфиге)

Рекомендуется задавать `excel_column` для фото‑слотов (позиционная запись в Excel):

```yaml
photo_slots:
  - slot: 1
    excel_column: "O"
    required: true
```

## Редактор конфигурации

Вкладка **«Редактор конфигурации»** позволяет править отдельные блоки `config.yaml` без ручного открытия файла в стороннем редакторе.

### Загрузка
- **Обзор…** — выбрать произвольный YAML-файл.
- **Загрузить текущий** — открыть тот же `config.yaml`, который использует программа (путь с вкладки «Общие настройки»).

### Редактируемые блоки (слева)
- **semantic** — постоянные значения для всех записей (выпадающие списки «ТБ», «Город» и т.д.).
- **column_mapping** — сопоставление полей справочника и колонок реестра (простой и расширенный формат с `ref`, `excel_column`, `required`).
- **package** — формирование имени ZIP-пакета (`name_columns`, `datetime_format`).
- **checklists — ПК**, **checklists — Монитор** и другие разделы — каждый тип оборудования отдельно (поля чек-листа с **`default`**, списки **`values`**, слоты фото, `aliases`).

Остальные разделы (`template`, `reference`, `checklist_templates`, `price_reference`, пути и т.д.) в этом редакторе **не изменяются** — они сохраняются в файле как есть.

> **Шаблоны чек-листа по модели** (`checklist_templates.yaml`) через этот редактор **не правятся** — используйте кнопки на вкладке «Чек‑лист» или отдельный текстовый файл.

### Порядок работы
1. Загрузите конфигурацию.
2. Выберите блок слева — справа откроется его YAML.
3. Отредактируйте текст и нажмите **«Применить»** — изменения попадут в память, синтаксис YAML и структура конфига будут проверены.
4. Нажмите **«Сохранить»** или **«Сохранить как…»** — запись в файл (только при корректной конфигурации).
5. Если сохранён **тот же файл**, что загружен в программе, будет предложено **перезагрузить конфигурацию** в интерфейсе.

При переключении блока с несохранёнными правками программа спросит: применить, отменить изменения или остаться на текущем блоке.

### Ограничения
- При сохранении PyYAML может изменить порядок ключей и отступы; **комментарии в YAML не сохраняются**.
- Элементы списков под ключом **`values:`** в редакторе выделяются **жирным тёмно-синим** шрифтом (тот же размер и шрифт, что у текста YAML).
- Для правки путей, `template` и прочих разделов по-прежнему используйте внешний редактор или вкладку «Общие настройки».

## Конфигурация `config.yaml` (ключевые разделы)

### template
```yaml
template:
  path: "C:/path/to/template.xlsx"
  sheet: 0
  header_row: 1
  start_row: 3
```
- `header_row`: строка заголовков (1-based)
- `start_row`: строка первой записи (данные) в реестре (1-based) — влияет на навигацию/обновление/удаление

### column_mapping: сопоставление данных из справочника в реестр
**Простой формат**:
```yaml
column_mapping:
  "Инв.номер": "Инв.номер"
```

**Расширенный формат** (рекомендуется):
```yaml
column_mapping:
  "Город":
    ref: "Город"        # колонка в справочнике
    excel_column: "A"   # колонка в реестре (буква или номер)
    required: true
```

#### Как работает `excel_column`
- `excel_column` можно задавать **буквой** (`"A"`, `"AB"`) или **номером** (`1`, `28`).
- При чтении/записи приложение использует `excel_column` в приоритете. Если он не задан — пытается работать по имени заголовка.

### Сохранение последних выбранных путей
Приложение сохраняет в `config.yaml` последние выбранные пути, чтобы при следующем запуске они подставлялись автоматически:
- `reference.path` — справочник Excel
- `price_reference.path` — справочник цен (для производителя/модели)
- `template.path` — шаблон Excel
- `output_path` — файл реестра
- `photos_dir` — папка фото

### checklist_templates: файл шаблонов чек‑листа
```yaml
checklist_templates:
  path: ''   # пусто — checklist_templates.yaml рядом с config.yaml
```

Содержимое файла шаблонов — см. раздел **«Шаблоны чек‑листа по модели»** выше. Путь можно указать абсолютным или относительным.

### price_reference: справочник цен
```yaml
price_reference:
  path: "C:/path/to/prices.xlsx"
  sheet: "Справочник"
  header_row: 3
```

Используется для полей производитель/модель и автоподстановки типа оборудования — см. раздел **«Справочник цен»**.

### semantic: общие (постоянные) значения
Это поля, которые задаются один раз и подставляются/записываются для каждой записи.

Пример:

```yaml
semantic:
  - key: city
    label: "Город"
    template_column: "Город"
    values: ["Москва"]
    required: true
```

#### `semantic[].excel_column` (приоритет над `template_column`)
Для `semantic` также можно задать `excel_column` — **куда именно записывать значение в реестр**.
Если `excel_column` указан, он имеет **приоритет** над `template_column`.

Пример:

```yaml
semantic:
  - key: city
    label: "Город"
    template_column: "Город"
    excel_column: "B"   # будет записано в колонку B, даже если заголовок/имя отличается
    values: ["Москва"]
```

### package: упаковка ZIP
```yaml
package:
  name_columns: ["Город", "Тип реализации"]
  datetime_format: "%Y-%m-%d_%H_%M"
```

Имя пакета: `Город_Тип реализации_YYYY-MM-DD_HH_MM.zip`.

#### Произвольный префикс/текст в имени пакета
В `package.name_columns` можно указывать не только названия колонок, но и **любой текст** (например, префикс `"Отчет"`).

Пример:

```yaml
package:
  name_columns: ["Отчет", "Город", "Тип реализации"]
  datetime_format: "%Y-%m-%d_%H_%M"
```

## Требования и ограничения при правке `config.yaml` (важно)

### YAML‑отступы
- YAML чувствителен к отступам. Рекомендуется **2 пробела** на уровень.
- Элемент списка всегда начинается с `-` на том же уровне, что и другие элементы списка.

### Строки и кавычки
- Пути с пробелами лучше писать в кавычках:

```yaml
price_reference:
  path: "C:/Users/.../Файл с пробелами.xlsx"
```

### Типовые ошибки
- Разрыв строки в середине `path` (получаются две строки) — приводит к ошибке парсинга.
- Неверный уровень отступа у `fields:`/`photo_slots:`/`aliases:` — приводит к ошибке парсинга.

### Рекомендация по изменениям
- После правки `config.yaml` нажмите **«Загрузить»** в интерфейсе (вкладка общих настроек), чтобы UI пересобрался.

## Буфер обмена
В полях ввода работают стандартные сочетания Qt:
- Ctrl+V / Shift+Insert — вставить
- Ctrl+C — копировать
- Ctrl+X — вырезать
- Ctrl+A — выделить всё
- Правый клик мыши — системное контекстное меню

## Развёртывание на Windows

Ниже — пошаговая инструкция для неопытного пользователя: от подготовки папки до ярлыка на рабочем столе.

### 1) Подготовка папки приложения
- Создайте папку, например: `C:\Apps\SaleMaker`
- Скопируйте туда все файлы проекта (в папке должны быть `main.py`, `config.yaml`, `requirements.txt` и т.д.)

### 2) Установите Python
1. Установите **Python 3.11** или **Python 3.12**.
   - Самый простой способ: **Microsoft Store**
   - Альтернатива: официальный установщик Python
2. Проверьте, что Python доступен:
   - Нажмите `Win+R` → введите `cmd` → Enter
   - Выполните:

```bat
python --version
```

Если команда не находится, переустановите Python и включите опцию **“Add python.exe to PATH”** (если она доступна).

### 3) Установка зависимостей (один раз)
1. Откройте **PowerShell**.
2. Перейдите в папку приложения (пример):

```powershell
cd C:\Apps\SaleMaker
```

3. Создайте виртуальное окружение:

```powershell
python -m venv .venv
```

4. Установите зависимости из `requirements.txt`:

```powershell
.\.venv\Scripts\python.exe -m pip install --upgrade pip
.\.venv\Scripts\python.exe -m pip install -r .\requirements.txt
```

### 4) Первый запуск (проверка)
В PowerShell (в папке приложения) выполните:

```powershell
.\.venv\Scripts\python.exe .\main.py
```

Если окно открылось — установка завершена.

### 5) Запуск двойным кликом (BAT-файл)
Чтобы запускать приложение как обычную программу, создайте BAT-файл.

1. Откройте **Блокнот** и вставьте (поменяйте путь, если папка другая):

```bat
@echo off
cd /d C:\Apps\SaleMaker
.\.venv\Scripts\pythonw.exe .\main.py
```

2. Сохраните файл в папку `C:\Apps\SaleMaker` с именем `Запуск SaleMaker.bat`  
   Важно: расширение должно быть **.bat**, а не `.txt`.

### 6) Ярлык на рабочем столе
1. Найдите `Запуск SaleMaker.bat` в папке приложения.
2. Нажмите правой кнопкой → **Отправить → Рабочий стол (создать ярлык)**.
3. (Опционально) Переименуйте ярлык на рабочем столе в **SaleMaker**.

### 7) (Опционально) Иконка ярлыка
Если у вас есть файл иконки `.ico`:
1. ПКМ по ярлыку на рабочем столе → **Свойства**
2. **Сменить значок…** → выберите `.ico`

## Развёртывание на Linux (Debian, Python 3.9–3.13)

Интерфейс построен на **PySide6**. Пакет **shiboken6** идёт в составе PySide6 — это обязательные биндинги Qt; убрать их без полной замены GUI (PyQt6, Tkinter) нельзя. SaleMaker **не использует** shiboken напрямую в коде.

**Не задавайте** `LD_LIBRARY_PATH=/usr/lib/...` — подхватится **системный Qt6**, несовместимый с Qt из wheel PySide6 (ошибка `Qt_6_PRIVATE_API`).

### 1) Системные пакеты (один раз)

```bash
sudo apt update
sudo apt install -y python3 python3-venv python3-pip \
  libgl1 libglib2.0-0 libxkbcommon0 libxcb-cursor0 libxcb-xinerama0 \
  libfontconfig1 libegl1 libopengl0
```

### 2) Виртуальное окружение

```bash
cd ~/Apps/SaleMaker
python3 --version

rm -rf .venv
python3 -m venv .venv
.venv/bin/pip install --upgrade pip
.venv/bin/pip install -r requirements.txt
.venv/bin/python check_env.py
```

Для **Python 3.9 + Anaconda** и ошибки `GLIBCXX_3.4.29`:

```bash
.venv/bin/pip install -r requirements-linux-py39-legacy.txt
SALEMAKER_USE_SYSTEM_LIBSTDCXX=1 ./run-linux.sh
```

Флаг `SALEMAKER_USE_SYSTEM_LIBSTDCXX` подгружает только **libstdc++.so.6**, не системный Qt.

### 3) Запуск

```bash
chmod +x run-linux.sh
./run-linux.sh
# или: .venv/bin/python main.py
```

### Типовые проблемы (Linux)

| Ошибка | Решение |
|--------|---------|
| `GLIBCXX_3.4.29 not found` | `requirements-linux-py39-legacy.txt`, venv от `/usr/bin/python3`, запуск с `SALEMAKER_USE_SYSTEM_LIBSTDCXX=1` |
| `Qt_6_PRIVATE_API` / `libQt6DBus` | Уберите `LD_LIBRARY_PATH` на `/usr/lib`; переустановите PySide6: `pip install -r requirements.txt --force-reinstall` |
| Anaconda в PATH | `deactivate`, используйте только `.venv/bin/python` |
| Нет камеры | `/dev/video0`, права пользователя, backend V4L2 |

## Типовые проблемы (Windows)
- **Ошибка “No module named …”** (в т.ч. `PySide6`): зависимости не установлены — повторите шаг 3.4 (`pip install -r requirements.txt`).
- **Не открывается камера**: выберите другую камеру в списке на вкладке «Фотографирование» и нажмите «Обновить».
